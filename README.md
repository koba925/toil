# toil

学習用の言語処理系について、実装しながら考えたり検証したりします。

## 狙い

[350行くらいのPythonで作るプログラミング言語実装超入門](https://zenn.dev/kb84tkhr/books/mini-interpreter-in-350-lines)（以下「350行言語」）のような書籍のネタにするための言語処理系を考える。

以下のような点で差異化する。

* 最初に評価器を作る。
  350行言語では、なぜ構文解析が出てくるのかうまく説明できなかった。
  最初に、コンピュータで処理しやすい形から入り、その後人間のための形を追加するという流れにする。
* 最初の評価器はとにかく小さく作り、最初の敷居を下げる。ここだけ読んでも意義があるようにする。
* 最終的な行数にはこだわらない。
* マクロを導入する。
* 簡易的なカスタム文法を導入する。
* ツリーウォーク型の処理系だけではなく、中間言語へのコンパイル＋仮想マシンの形でも実装する。

以下の点は継続する。

* できるだけ平易なコードを書く。
* できるだけ予備知識を減らす。
* エラー処理や性能については問わず、言語処理系の原理の学習に的を絞る。

## 開発方針

* 言語処理系の原理を学習することを主要な目的とし、そのための最小限の実装を行う。
　* 実用上の課題よりも仕様・実装のシンプルさを優先する（末尾のセミコロン・カンマを許さない等）。
* 細かいステップでインクリメンタルに開発する。
* 標準・サードパーティ問わずライブラリの利用はできるだけ控える。ただしテスト用にpytestのみ導入する。
* 「後で必要になるから」と必然性のないコードはできるだけ入れないようにし、必要になったときに初めてコードを追加する。ただし、以下については先を見越してコードを書く場合もある。
  * 最終系のプログラムの骨組みを意識して書く。
  * 状態を持たず関数で実装できる場合でもクラス＋メソッドで書く。
    * 関心の分離を明確に表すため
    * 名前空間をクリーンに保つため
    * 一貫性を持つため
  * その他、あとで大量の書き直しが発生すると予想される場合。

## 現在の言語機能

* **データ型**: 整数、ブーリアン、None、関数（クロージャ）、配列、文字列
* **制御構文**: `if-then-elif-else-end`, `while-do-end`
* **変数**: `:=`による定義、`=`による代入、静的スコープ（クロージャ）
* **演算子**:
  * 四則演算: `+`, `-`, `*`, `/`, `%`
  * 比較演算: `==`, `!=`, `<`, `>`, `<=`, `>=`
  * 論理演算: `and`, `or`（短絡評価）, `not`
* **配列**:
  * `arr()`による生成、`[]`によるインデックスアクセス（負数インデックスも可）
  * `len`, `push`, `pop`, `slice` などの組み込み関数
* **文字列**:
  * シングルクォート (`'...'`): エスケープなしの生文字列
  * ダブルクォート (`"..."`): `\n`, `\"`, `\\` などのエスケープシーケンスが利用可能
  * `str`, `int`, `chr`, `ord`, `join` などの組み込み関数
* **標準ライブラリ**:
  * `toil`言語自身で実装されたライブラリ群
  * 高階関数: `map`, `filter`, `reduce`
  * ユーティリティ: `range`, `zip`, `enumerate`, `reverse`, `first`, `rest`, `last`

## 書籍の構成案

書籍の全体構成は以下を想定している。
どこで読むのをやめてもまとまりのある学びを得られるようにする。
第４部までが通常の入門で、その先は趣味レベル。

* 第１部：基本の評価器
  Pythonの配列・タプルを用いたASTでごく小さい評価器を作成します。
* 第２部：基本の字句解析・構文解析
  第一部の評価器に字句解析・構文解析を追加して、基本のインタプリタを作成します。
* 第３部：機能の拡張
  配列・文字列・パターンマッチなど、やや高度な機能を追加します。
* 第４部：中間言語インタプリタ
  第３部までの字句解析・構文解析はそのままに、中間言語へのコンパイルと仮想マシンを使ったインタプリタを作成します。
* 付録１：基本の評価器で評価器を実装
  第１部の評価器を使い、クロージャでcons・car・cdrを実装してセルフ評価器を作成します。
* 付録２：別言語
　第２部のインタプリタの字句解析・構文解析を書き直して別言語にします。
* 付録３：マクロとカスタム文法
  第３部のインタプリタにマクロとカスタム文法を追加して独自の構文を追加できるようにします。
* 付録４：中間言語インタプリタでもマクロとカスタム文法
  第４部のインタプリタに、事前に展開を済ませる方式でマクロを実装します。
* 付録５：末尾呼び出し最適化
  付録４のインタプリタに末尾呼び出し最適化を実装し、繰り返しを再帰で書けるようにします。
* 付録６：継続
  継続を実装して制御構造もマクロで書けるようにし、簡単なジェネレータや並列化を実装します。
* 付録７：toilでtoilを書く１
  付録３のインタプリタでセルフインタプリタを作成します。サブセットにするかも。
* 付録８：toilでtoilを書く２
  付録６のインタプリタでセルフインタプリタを作成します。仮想マシンはあらたに実装はせず、付録６の仮想マシンにコンパイルした中間コードを渡す形式とします。

## UFCSに関する補足

UFCSでは、以下の３つのケースを区別する必要がある。
これらは構文解析の時点では区別できないので、評価時に区別する。

* ケース１：関数を記録する名前空間的なコンテナ＋関数
  * 定義：gcd := { recur: func a, b do <return GCD of a and b> end }
  * 呼び出し：gcd.recur(24, 36)
  * 意味：gcdは単なる関数を記録する名前空間的なdict型のコンテナで、gcd.recurは単なる関数であり、gcdを引数としては受け取らない
　* 判別方法：gcdにはrecurというキーがあり、関数にバインドされている。第一引数はselfではない

ケース２：オブジェクト＋メソッドとして
  * 定義： dog := { cry: func self, sound do <do something with self> end }
  * 呼び出し：dog.cry("bow")
  * 意味：dogはオブジェクトであり、cryは第一引数にオブジェクトを取る関数で、呼び出し時にはselfにdogがバインドされる
  * 判別方法：docにはcryというキーがあり、関数にバインドされている。第一引数がself

* ケース３：グローバル関数の呼び出し（UFCS: Unified Function Call Syntax）
  * 定義： d := 任意のdict
  * 呼び出し： d.filter(f)
  * 意味： dは単なるdict型のデータで、filterはdict型のデータを第一引数に取る関数。呼び出し時にはdがfilterの第一引数にバインドされる
  * 判別方法： dにはfilterというキーが存在しない（または、存在してもバインドされているものがfunc以外）
  * 補足：ケース１のgcdモジュールにfilterというキーで関数が定義されているとグローバル関数ではなくgcd.filterが呼ばれる

## コーディング規約

* 書籍で説明する予定であることと、コードを短く見通しよくしたいことからコードにはコメントを入れない。
  * コメントを入れなくても読みやすいコードになるよう心がける。
* エラーチェックは最小限とする。Pythonのエラーで止まっても何が起きたかわかれば問題ないものとする。
* エラーはassertで上げる。エラーメッセージは f"{メッセージ} @ {関数}() : {エラーとなった値}"とする。
* エラーチェックをつける代表的なケースは以下。
  * matchで適切なcaseがなかった場合。
  * 型をチェックしないと警告が出る場合。
* 変数名や関数名、エラーメッセージなどは自然で簡潔な英語で短く書く。
　* 短縮形も利用する（evaluate→eval、identbol→ident、expression→exprなど）
* 引用符はダブルクォーテーションを使う。
  * toilのソースコードは三重ダブルクォーテーションで囲んで記述する（toilのソースコードに引用符を使うようになることを見越して）。
* 見通せる範囲の情報量を増やすため、PEP-8にしたがえば改行を入れるところでも入れない場合がある。
  特に、トップレベルの定義の間は基本的には1行のみ（大きな区切りでは2行にすることもある）。
* 0や1はFalse・Trueやその他の特別な値と混乱する場合があるため、テストで使う数は2から始める。
* テストメソッドはできるだけコードとの対応がつけやすいように並べる。
* toil.py、toil_on_toil.pyのメインセクションには実行例となるコードを書く。
  * 実行例がprint()で出力するときは出力結果を `# -> ` に続けてコメントする。
* Interpreterクラスの実行例では内部表現も試すが、テストでは内部表現はテストしない。
  * テストではInterpreter.walk()メソッドを使う。どうしてもgo()でテストできない場合のみscan()、parse()、evaluate()でテストするようにする。
* 型ヒントは要所のみにつける。
  * 型チェックというよりも、意図や仕様を明確にするのを目的とする。
  * 警告が出るのを抑制するためにつける場合もある。
